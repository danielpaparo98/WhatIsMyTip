# Loading real AFL data

> Phase 7 of the local Docker stack вЂ” how to populate the database with
> real AFL data scraped from public sources.

## TL;DR

```bash
# 1. Generate CSVs into ./data/  (10-30 min depending on scope)
./scripts/make-data.sh

# 2. Bring up the full stack вЂ” the init-data container auto-loads the CSVs
./scripts/dev.sh up --logs
```

That's it.  The `init-data` Docker service bind-mounts `./data/` and
runs [`scripts/migrate_and_seed.py --from-csv`](../backend/scripts/migrate_and_seed.py:1)
on first start, so the API never boots against an empty database.

## Scope decision (2020 вЂ“ current season)

The default scope is **5 most recent completed seasons** (2020 вЂ“ current
at time of writing).  This balances two competing concerns:

| Concern | Why it matters |
|---|---|
| Enough history for Elo / Form / HomeAdvantage / Value / Matchup models | These models need в‰Ґ 3 seasons to stabilise; 5 is comfortable. |
| Bounded scrape time | 5 seasons в‰€ 1,000 games Г— 3 web requests each в‰€ 30 minutes. |
| Bounded CSVs | Player stats alone are ~150,000 rows across 5 seasons. |

To change the scope:

```bash
# Bash вЂ” just two seasons, fastest
SEASONS="2024 2025" ./scripts/make-data.sh

# Bash вЂ” full history since 2015
SEASONS="2015 2016 2017 2018 2019 2020 2021 2022 2023 2024 2025" ./scripts/make-data.sh

# PowerShell
$env:SEASONS = "2024 2025"; .\scripts\make-data.ps1
```

## Data sources

The scraper pulls from three public sources, each chosen for the data
it provides and the rate limits it tolerates.

| Source | What we get | Rate limit | CSV file(s) |
|---|---|---|---|
| [AFL Tables](https://afltables.com/afl) | Historical match results, season game listings, per-match player stats, advanced match metadata | Generous (no key required) | `season_games.csv`, `match_details.csv`, `players.csv`, `player_match_stats.csv` |
| [FootyWire](https://www.footywire.com) | Current injury list | Generous | `injuries.csv` |
| [Open-Meteo](https://open-meteo.com) | Historical + forecast weather for each game venue/date | 10,000 requests/day | `match_weather.csv` |

The scraper is rate-limited via:
* **AFL Tables** вЂ” sequential fetches with ~1 req/sec (no formal limit,
  but the site throttles aggressive scrapers).
* **FootyWire** вЂ” 1 req/run (the injury list is a single page).
* **Open-Meteo** вЂ” Redis-backed sliding-window rate limiter (see
  [`packages/shared/squiggle/client.py`](../backend/packages/shared/squiggle/client.py:1)
  for the pattern; the same `_RedisRateLimiter` could be reused if
  Open-Meteo throttling becomes an issue).

## How the loader finds the data

`scripts/migrate_and_seed.py --from-csv` (used by `init-data`) calls
[`find_csv_seed_dir()`](../backend/scripts/migrate_and_seed.py:148)
which walks these candidate locations in order, taking the first
that contains at least one `.csv` file:

1. `./data/`                     (what `make-data.sh` produces)
2. `./backend/seed_data/`        (the synthetic seed CSVs)
3. `./backend/data/`             (legacy location)
4. `./scripts/data/`             (dev last-resort)

So `./data/` always wins when present.

## When to re-scrape

| Trigger | Action |
|---|---|
| New season starts (March) | Add the new season to `SEASONS` and re-run. |
| New AFL season mid-year | Same вЂ” append the new year. |
| Bug in upstream HTML | Re-run; the scraper overwrites the per-season CSVs. |
| `injuries.csv` is stale | Re-run with just `--table injuries`. |

The CSVs are idempotent вЂ” re-running `make-data.sh` overwrites the
per-season files.  The Postgres loader uses `ON CONFLICT DO NOTHING`
for players and stats, so re-running is safe.

## Known limitations

* **No player advanced stats** вЂ” the player_form model also wants
  `player_advanced_stats` (TOG%, metres gained, etc.).  These are
  available from FootyWire but require per-player scraping (one
  HTTP call per player) which is much slower.  Out of scope for
  Phase 7; tracked for Phase 8.
* **No team selections** вЂ” `get_team_selections` from FootyWire is
  not wired into the scraper.  Only injuries from FootyWire are
  pulled in Phase 7.
* **No live game results** вЂ” the scraper only pulls historical data.
  Live game results are pulled by the
  [`daily-sync`](../backend/app/cron/daily_sync.py:1) cron job via
  the Squiggle API.
* **Scraper runs sequentially** вЂ” a single AFL Tables HTTP call per
  game.  At ~1 req/sec, a 5-season scrape takes ~20-30 minutes.
  Parallelising the fetcher is tracked for Phase 8.

## Re-scoping for future maintainers

When a new maintainer picks this up and wants to add 2026 data:

```bash
# Edit scripts/make-data.sh (or pass SEASONS env var)
SEASONS="2020 2021 2022 2023 2024 2025 2026" ./scripts/make-data.sh

# Or just append to the default in the script
```

The CSVs are versioned by the season directory (e.g. `data/2026/`)
so old and new data coexist.  Re-running `./scripts/dev.sh reset &&
./scripts/dev.sh up` reloads everything from the new CSVs.
